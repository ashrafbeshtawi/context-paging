Context Paging: Virtual Memory for AI Agent Context Windows

Ashraf Beshtawi
March 2026

Abstract

Large language model (LLM) agents operating over extended interactions inevitably encounter a fundamental constraint: the finite context window. Current mitigation strategies — truncation, summarization, and retrieval-augmented generation — each sacrifice fidelity, determinism, or agent autonomy. We introduce Context Paging, an approach that applies the operating system virtual memory paging model to LLM context management. In this framework the context window is treated as physical RAM, a filesystem serves as backing store, and a page table maintained in the system prompt gives the agent a persistent index of everything it has stored. The agent is equipped with seven paging tools — page_out, page_in, page_table, page_update, page_free, page_move, and page_merge — enabling it to explicitly manage its own memory: swapping context to disk when capacity is scarce and restoring it on demand. We present a proof-of-concept implementation built with TypeScript and the Vercel AI SDK, supporting seven LLM providers, and validated by a suite of 69 unit tests. Context Paging is lossless, deterministic, and agent-controlled, offering a complementary mechanism to existing approaches.

1. Introduction

AI agents powered by large language models are increasingly deployed in long-running, multi-turn conversations: debugging sessions that span hundreds of messages, research tasks that accumulate large bodies of intermediate results, and collaborative workflows where context from hours ago remains relevant. In each of these scenarios the agent's context window — the fixed-size buffer of tokens the model can attend to at any given inference step — becomes the binding constraint.

When the conversation exceeds the context window, something must give. The three dominant strategies each carry significant trade-offs:

  • Truncation discards the oldest messages outright. Context is lost permanently; the agent cannot recover it.
  • Summarization compresses older context into a shorter representation. This is inherently lossy: specific details, code snippets, exact error messages, and nuanced reasoning are discarded in favor of a compressed gist.
  • Retrieval-Augmented Generation (RAG) stores context externally and retrieves relevant fragments via semantic search at query time. Retrieval quality is query-dependent and non-deterministic; relevant context may be missed if the retrieval query is poorly formed.

Critically, in all three approaches the agent has no explicit control over what is retained and what is discarded. The system makes the decision on its behalf — or the decision is made by the architecture before the agent is even invoked.

This paper proposes Context Paging, a method that gives the agent direct, tool-mediated control over its context lifecycle. The core insight is borrowed from operating systems: virtual memory paging. Just as an OS transparently swaps memory pages between RAM and disk to present processes with an address space larger than physical memory, Context Paging lets an agent swap conversation segments between its active context window and persistent storage, maintaining a page table as a compact index of everything stored.

2. Background

2.1 LLM Context Windows and Their Limitations

A large language model processes input as a sequence of tokens within a fixed-length context window. Modern models offer windows ranging from 8K tokens (earlier GPT-4 variants) to 200K tokens (Claude 3.5) and beyond (Gemini 1.5 Pro at 1M tokens). Despite these increases, the context window remains finite, and its effective utilization degrades well before the hard limit: studies have shown that models exhibit a "lost in the middle" phenomenon [1], where information placed in the center of a long context is recalled less reliably than information near the beginning or end.

For agentic workloads — where the conversation may span dozens of tool calls, branching subtasks, and iterative refinements — the context window fills rapidly. Each tool call and its result, each intermediate reasoning step, and each user clarification consumes tokens. A debugging session of moderate length can easily reach 50,000–100,000 tokens, with much of that content being intermediate work that is no longer actively needed but may become relevant again later.

2.2 Operating System Virtual Memory as an Analogy

The operating system's virtual memory subsystem solves an analogous problem. Physical RAM is finite, but processes require address spaces that may exceed available memory. The OS resolves this through demand paging:

  • Physical memory (RAM) is divided into fixed-size frames.
  • A process's virtual address space is divided into pages.
  • A page table maps virtual pages to physical frames (or marks them as swapped to disk).
  • When a process accesses a page not currently in RAM, a page fault triggers the OS to load it from disk.
  • When RAM is full, the OS selects a page to swap out (evict to disk), freeing a frame for the incoming page.

The mapping from this model to LLM context management is direct:

Table 1. Mapping between OS virtual memory and Context Paging.
OS Concept Context Paging Analogue
Physical RAMActive context window
Disk / swap spaceFilesystem storage (pages/ directory)
PageA saved unit of conversation context
Page tableLightweight index injected into the system prompt
Page fault (swap in)page_in tool call
Swap out / evictionpage_out tool call
Free / deallocationpage_free tool call

The key difference is agency: in an OS, the kernel manages paging transparently. In Context Paging, the LLM agent itself decides what to page out and when to page in, using explicit tool calls. This gives the agent awareness of and control over its own memory constraints — a form of metacognitive memory management.

3. Approach — Context Paging

3.1 Architecture

The Context Paging system consists of three layers: (1) the LLM agent with its standard conversational capabilities; (2) a set of seven paging tools exposed to the agent via function calling; and (3) a storage backend that persists pages to the filesystem.

Table 2. The seven paging tools.
Tool OS Analogy Purpose
page_out Swap out Save context to disk, remove from active window
page_in Swap in / page fault Load a page's content back into active context
page_table Page table query View the index of all stored pages and their status
page_update — Modify a page's metadata (title, summary) or content
page_free Free / dealloc Permanently delete a page from disk
page_move — Reorganize page hierarchy (re-parent a page)
page_merge — Consolidate multiple pages into one

3.2 The Paging Lifecycle

A typical paging cycle proceeds as follows:

  1. The agent accumulates context through normal conversation: user messages, assistant responses, tool calls and results.
  2. At a decision point — the end of a subtask, a topic switch, or an explicit user request — the agent calls page_out with a title, summary, content, and a swap_count parameter indicating how many recent messages to remove from context.
  3. The system writes the content to disk as a numbered page directory, removes the specified messages from the conversation array, and inserts a compact placeholder: [Paged out → Page N: "title" — summary].
  4. The next LLM inference call sees a smaller context with the placeholder, plus an updated page table in the system prompt.
  5. When the agent later needs that context, it calls page_in. The full content is loaded from disk and appended to the message array. 
            ┌──────────────────────────────────────┐
            │       Active Context Window (RAM)      │
            │                                        │
            │  Recent messages + page placeholders   │
            └──────────┬──────────────▲──────────────┘
                       │              │
                  page_out        page_in
                  (swap out)      (swap in)
                       │              │
            ┌──────────▼──────────────┴──────────────┐
            │         Disk Storage (pages/)           │
            │                                         │
            │  Page 1: "Auth debugging"    [swapped]  │
            │  Page 2: "API design"       [resident]  │
            │  Page 3: "Test results"      [swapped]  │
            └─────────────────────────────────────────┘
Figure 1. The paging lifecycle. Context is swapped between the active context window and disk storage via tool calls.

3.3 Page Table Injection

A critical design decision is how the agent maintains awareness of its stored pages. In Context Paging, the current page table is injected into the system prompt before every LLM call. This ensures the agent always sees an up-to-date index of all pages — their IDs, titles, summaries, timestamps, and residency status — without those pages consuming significant context space. The page table serves the same role as an OS page table: a compact mapping that lets the agent know what it has stored and where to find it.

This approach is implemented in the agent's core loop: 

// Inject live page table into system prompt
const pageTable = await handlePageTable({});
const systemWithPageTable = `${SYSTEM_PROMPT}

## Current Page Table
${pageTable}`;

3.4 Provider-Agnostic Design

Context Paging is implemented using the Vercel AI SDK, which provides a unified interface across multiple LLM providers. The paging logic is entirely independent of the underlying model; any provider that supports tool calling can use the system. The proof-of-concept implementation supports seven providers: Anthropic (Claude), OpenAI (GPT-4o), Google (Gemini), Mistral, xAI (Grok), Amazon Bedrock, and Azure OpenAI. The provider and model are selected at runtime via environment variables, with no code changes required.

3.5 Storage Format

Pages are persisted as directories on the filesystem, enabling hierarchical organization through nesting: 

pages/
  _counter.json              # Auto-incrementing page ID tracker
  1/                         # Page directory
    meta.json                # { id, title, summary, created_at, updated_at, is_resident }
    content.md               # Full page content (markdown)
    3/                       # Nested child page
      meta.json
      content.md
  2/
    meta.json
    content.md

Each page's metadata is stored separately from its content, allowing the page table to be constructed by reading only the lightweight meta.json files. Content is stored as Markdown, preserving formatting, code blocks, and structured data without transformation. The is_resident flag in metadata tracks whether a page's content is currently present in the active context.

4. Proof of Concept Implementation

4.1 Technology Stack

The proof-of-concept is implemented in TypeScript and structured as a modular Node.js application:

4.2 Runtime Behavior

The agent operates as an interactive CLI application with streaming responses. When the agent calls page_out, the swap is performed immediately: the specified number of messages are removed from the conversation array and replaced with a placeholder. The user sees a context indicator showing the current number of resident messages. An optional debug mode displays tool call arguments and results in real time.

4.3 Swap Mechanics

The swap-out operation is implemented as a pure function over the message array: 

function swapOut(
  messages: ModelMessage[],
  pageId: number,
  title: string,
  summary: string,
  swapCount?: number
): ModelMessage[] {
  if (!swapCount || swapCount <= 0) return messages;
  const keepCount = Math.max(0, messages.length - swapCount);
  const result = messages.slice(0, keepCount);
  result.push({
    role: "assistant",
    content: `[Paged out → Page ${pageId}: "${title}" — ${summary}]`,
  });
  return result;
}

This design ensures that the swap operation is deterministic and testable: given the same inputs, it always produces the same output. The message array is treated as immutable; a new array is returned rather than mutating the original.

4.4 Validation

The implementation is validated by a comprehensive test suite of 69 unit tests built with Vitest. The tests cover all seven tool operations, edge cases (circular nesting prevention, missing pages, recursive deletion), the swap mechanics, storage operations, and the page table formatter. All 69 tests pass.

5. Comparison with Existing Approaches

Context Paging is not a replacement for existing context management strategies but a complementary mechanism. The following table compares the four approaches across key dimensions:

Table 4. Comparison of context management approaches.
Dimension Truncation Summarization RAG Context Paging
Lossless No No Partial Yes
Agent-controlled No No No Yes
Deterministic Yes No No Yes
Recoverable No No Yes Yes
No extra infrastructure Yes Yes No Yes
Context-aware retrieval N/A N/A Partial Yes

Lossless. Truncation permanently discards context. Summarization compresses it, losing detail. RAG stores original documents but retrieval may return only fragments. Context Paging stores the full content and restores it exactly as it was.

Agent-controlled. Only Context Paging gives the agent explicit decision-making authority over what to store, when to store it, and when to recall it. In all other approaches, the mechanism operates outside the agent's control.

Deterministic. When the agent calls page_in(3), it always gets exactly the content of Page 3. There is no embedding similarity threshold, no summarization variability, and no retrieval ranking to introduce uncertainty.

Context-aware retrieval. Because the agent sees the page table at every turn, it can make informed decisions about which pages are relevant based on the full conversational context, not just a query embedding.

6. Future Work

6.1 Benchmarking

The most immediate priority is rigorous benchmarking. We plan to evaluate Context Paging against standard long-context benchmarks, comparing agent performance with and without paging enabled. Key metrics include:

  • Task completion accuracy over conversations that exceed the context window length.
  • Context recall fidelity — how accurately the agent retrieves and applies previously paged-out information.
  • Token efficiency — total tokens consumed per task, measuring whether paging reduces overall cost by keeping the active context lean.

6.2 Paging Trigger Strategies

A critical open question is when the agent should page out. The current implementation relies on the agent's autonomous judgment, guided by system prompt instructions. We identify several trigger strategies worth investigating:

  • Context-percentage threshold. Automatically prompt the agent to page out when the context window reaches a configurable percentage of capacity (e.g., 75%).
  • Free-form (autonomous). Let the model decide entirely on its own, based on its understanding of the conversation flow.
  • Message-count based. Trigger a paging prompt every N messages.
  • Token-budget based. Page out when the remaining token budget drops below a threshold, ensuring the agent always retains enough capacity for its next response.
  • Hybrid approaches. Combine multiple signals — for instance, token budget as a hard constraint with autonomous paging as a soft preference.

6.3 Automatic vs. Manual Paging

The current design is fully manual: the agent must explicitly invoke paging tools. An alternative is automatic paging, where the system intercepts the context before each LLM call and transparently pages out older content. This mirrors the OS model more closely (where processes are unaware of paging) but sacrifices the agent's ability to make informed decisions about what to retain. A hybrid approach — automatic paging as a safety net with agent-initiated paging as the primary mechanism — may offer the best of both worlds.

6.4 Multi-Agent Paging

In multi-agent architectures, multiple agents may need to share context. Context Paging's filesystem-based storage naturally supports this: agents can share a page directory, reading each other's pages. Future work should explore shared page tables, concurrent access patterns, and page-level locking semantics for agent swarms collaborating on a shared task.

6.5 Compression: Summarization Before Paging

Context Paging and summarization are not mutually exclusive. A promising direction is to combine them: before paging out, the agent could summarize the content and store both the summary (for quick reference) and the full content (for lossless recovery). This would reduce disk usage and potentially allow a two-tier retrieval strategy — check the summary first, page in the full content only if needed.

7. Conclusion

Context Paging applies one of the oldest and most successful ideas in computer science — virtual memory paging — to one of the newest challenges in AI systems: managing finite context windows in long-running agent interactions. By giving the agent explicit tools to swap context between its active window and persistent storage, and by maintaining a page table that provides constant awareness of stored state, Context Paging enables lossless, deterministic, and agent-controlled memory management.

The approach is complementary to existing strategies. It does not replace RAG for knowledge retrieval over large corpora, nor does it eliminate the value of summarization for creating concise representations. Rather, it fills a gap: giving agents the ability to manage their own conversational memory with the same precision and control that operating systems bring to process memory management.

The proof-of-concept implementation demonstrates that the approach is practical, provider-agnostic, and straightforward to implement with existing tool-calling infrastructure. The key open questions — optimal trigger strategies, automatic vs. manual paging, and multi-agent coordination — define a productive research agenda. Context Paging is a step toward agents that are not merely consumers of context but active managers of their own cognitive resources.
